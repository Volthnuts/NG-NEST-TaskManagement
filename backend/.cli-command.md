# 1. คำสั่งสร้าง Core Modules (ระบบหลัก)

แนะนำให้ใช้คำสั่ง `res` (resource) เพราะมันจะถามเพื่อสร้างระบบ CRUD (REST API) และสร้างไฟล์ DTO มาให้พร้อมใช้งานทันทีครับ

```bash
# สร้างโมดูลระบบสมาชิก (Authentication)
nest g res modules/auth --no-spec

# สร้างโมดูลจัดการข้อมูลผู้ใช้งาน (Users)
nest g res modules/users --no-spec

# สร้างโมดูลจัดการโปรเจกต์ (Projects)
nest g res modules/projects --no-spec

# สร้างโมดูลจัดการงาน (Tasks)
nest g res modules/tasks --no-spec
```

---

# 2. คำสั่งสร้างส่วนกลาง (Common Folder)

ใช้สำหรับสร้างระบบฐานข้อมูล, ตัวกรองความปลอดภัย, และตัวจัดฟอร์แมตข้อมูลกลางครับ

---

## ก. ระบบฐานข้อมูล (Prisma)

```bash
# สร้าง Module สำหรับ Prisma
nest g mo common/prisma

# สร้าง Service สำหรับ Prisma (เชื่อมต่อ DB)
nest g s common/prisma --no-spec
```

---

## ข. ระบบรักษาความปลอดภัย (Guards)

```bash
# สร้าง Guard เช็กการล็อกอิน (JWT)
nest g gu common/guards/jwt-auth --no-spec

# สร้าง Guard เช็กสิทธิ์ผู้ใช้ (Roles / RBAC)
nest g gu common/guards/roles --no-spec
```

---

## ค. ตัวดักจับและแปลงข้อมูล (Interceptors)

```bash
# สร้าง Interceptor สำหรับจัดฟอร์แมตข้อมูลส่งกลับหน้าบ้าน
nest g itc common/interceptors/transform --no-spec

# สร้าง Interceptor สำหรับบันทึก Log การทำงาน
nest g itc common/interceptors/logging --no-spec
```

---

## ง. ป้ายกำกับทางลัด (Decorators)

```bash
# สร้าง Custom Decorator (เช่น @GetUser)
nest g d common/decorators/get-user --no-spec

nest g d common/decorators/roles --no-spec
```

---

## จ. แม่พิมพ์โครงสร้างข้อมูล (Interfaces)

```bash
# สร้าง Interface สำหรับระบุข้อมูลใน Token และ Request
nest g interface common/interfaces/jwt-payload --no-spec

nest g interface common/interfaces/request-with-user --no-spec
```

---

# สรุปความหมายของออปชัน (Flags)

| คำสั่ง      | ความหมาย             |
| ----------- | -------------------- |
| `g`         | generate (สั่งสร้าง) |
| `res`       | resource             |
| `mo`        | module               |
| `s`         | service              |
| `co`        | controller           |
| `gu`        | guard                |
| `itc`       | interceptor          |
| `d`         | decorator            |
| `interface` | interface            |

```bash
--no-spec
```

ไม่สร้างไฟล์ทดสอบ (`.spec.ts`) เพื่อไม่ให้โครงสร้างสับสนในช่วงแรก

# 1. Guards (ผู้รักษาความปลอดภัย)

## หน้าที่

ตัดสินใจว่า Request นี้ “มีสิทธิ์เข้าเมือง” หรือไม่ ก่อนจะไปถึงตัว Controller

## ในโปรเจกต์นี้

### `jwt-auth.guard.ts`

เช็กว่าคนยิง API ล็อกอินหรือยัง?

* มี Token หรือไม่
* Token ปลอมหรือหมดอายุหรือไม่

---

### `roles.guard.ts`

เช็กว่าคนล็อกอินมี “สิทธิ์” พอไหม

ตัวอย่าง:

* จะลบโปรเจกต์ → ต้องเป็น `OWNER`
* ถ้าเป็น `MEMBER` ทั่วไป → Guard จะบล็อกไว้และส่ง `403 Forbidden`

---

# 2. Decorators (ป้ายกำกับทางลัด)

## หน้าที่

สร้าง “คำสั่งสั้นๆ” (ที่ขึ้นต้นด้วย `@`) เพื่อดึงข้อมูลจาก Request ออกมาใช้ง่ายๆ หรือใช้ระบุคุณสมบัติพิเศษ

## ในโปรเจกต์นี้

### `get-user.decorator.ts`

ปกติข้อมูลคนล็อกอินจะอยู่ใน Request ที่ซับซ้อน
พอมีตัวนี้ คุณแค่เขียน:

```ts
@GetUser()
```

ใน Controller ก็จะได้ข้อมูล User (`id`, `email`) มาใช้ทันที

---

### `roles.decorator.ts`

เอาไว้แปะบนหัวฟังก์ชันเพื่อบอกว่า “เส้นนี้เฉพาะ Admin นะ”

ตัวอย่าง:

```ts
@Roles('ADMIN')
```

---

# 3. Interceptors (ตัวดักจับและแปลงร่าง)

## หน้าที่

ดักจับข้อมูล “ขาก่อนเข้า” หรือ “ขาก่อนออก” เพื่อแก้ไขหรือเสริมข้อมูลบางอย่าง

## ในโปรเจกต์นี้

### `transform.interceptor.ts`

สำคัญมาก เพราะใช้จัดระเบียบข้อมูล Response ให้ Frontend อ่านง่าย

ตัวอย่าง:

จากเดิม:

```json
{
  "id": 1,
  "title": "Task A"
}
```

แปลงเป็น:

```json
{
  "success": true,
  "data": {
    "id": 1,
    "title": "Task A"
  },
  "message": "Success"
}
```

ทำให้ทุก API มีโครงสร้างเหมือนกันทั้งหมด

---

### `logging.interceptor.ts`

ใช้บันทึก Log ลง Terminal เช่น

* มีคนยิง API เส้นไหน
* ใช้เวลาประมวลผลกี่ ms
* Error หรือไม่

ช่วย Debug ระบบได้ง่ายมาก

---

# 4. Interfaces (แม่พิมพ์โครงสร้าง)

## หน้าที่

นิยาม “หน้าตาข้อมูล” ในเชิง TypeScript เพื่อให้โปรแกรมรู้ว่าตัวแปรนี้มีฟิลด์อะไรบ้าง

ช่วย:

* ลด Bug
* Auto Complete ดีขึ้น
* ป้องกันการส่งข้อมูลผิดประเภท

## ในโปรเจกต์นี้

### `jwt-payload.interface.ts`

กำหนดว่าใน JWT Token จะมีข้อมูลอะไรบ้าง เช่น

```ts
{
  id: string;
  email: string;
  role: string;
}
```

---

### `request-with-user.interface.ts`

บอก TypeScript ว่า Request ของระบบเราจะมี `user` แปะมาด้วย

จากเดิม:

```ts
req.user // TypeScript Error
```

หลังใช้ Interface:

```ts
req.user.id
req.user.email
```

จะไม่ขึ้นเส้นแดง

---

# 5. Prisma / Database (ท่อเชื่อมต่อฐานข้อมูล)

## หน้าที่

เป็นศูนย์กลางการเชื่อมต่อฐานข้อมูลของทั้งระบบ

## ในโปรเจกต์นี้

### `prisma.service.ts`

ทำหน้าที่:

* Connect Database
* Disconnect Database
* เป็นตัวกลางยิง Query ไปยัง PostgreSQL / MySQL

ตัวอย่างการใช้งาน:

```ts
constructor(private prisma: PrismaService) {}
```

จากนั้นสามารถเรียก:

```ts
this.prisma.tasks.findMany()
```

ได้ทันที

---

# สรุป Workflow ของระบบ

```text
Request เข้ามา
    ↓
Guard เช็กสิทธิ์ / JWT
    ↓
Interceptor (ขาเข้า) บันทึก Log
    ↓
Controller รับ Request
    ↓
Decorator ดึงข้อมูล User
    ↓
Service ทำ Business Logic
    ↓
Prisma ติดต่อ Database
    ↓
ได้ข้อมูลกลับมา
    ↓
Interceptor (ขาออก) จัดรูปแบบ Response
    ↓
ส่งกลับ Angular / Frontend
```
